# Agent 通用运行框架

## 核心理念

本项目并非传统意义上的“Agent 使用外部工具进行交互”的框架。我们定义的 Agent 是：

> **大部分时间内只处理信息，而不直接操作工具**。

Agent 的核心能力是**思考**，然后才可能行动。因此，框架的设计围绕以下几个简单直觉展开：

1. **Agent 需要思考才能行动** – 任何外部输出都必须经过 LLM 的推理，没有硬编码的“工具调用”捷径。
2. **Agent 应该有简单的长期记忆** – 基于内部键值存储（KV）的访问模型，不依赖外部文件系统或数据库。
3. **Agent 可以独立完成任务，但框架的真正价值在于管理大量 Agent 的协作** – 有序、可控、可扩展。

## 设计原则

- **与特定编程语言解耦** – Agent 的行为描述不绑定任何具体语言，只依赖对 LLM 通用模式的抽象。
- **几乎没有“工具”抽象** – 我们不把外部能力包装成“工具”。系统只保留对 LLM 原生能力（文本生成、结构化输出）的抽象。
- **保持 LLM “语言运算”的纯洁性** – 框架本身不内置任何工具支持（如文件读写、网络请求、Shell 执行）。所有外部交互必须通过 **IO 接口** 进行，类似于 CPU 的 **PIO（程序控制输入输出）** 模式：Agent 通过读写特定的内存映射区域来触发外部操作，而不是直接调用工具。

## 与外部系统的交互

- 本项目不提供任何自带工具（如搜索、计算器、数据库）。
- 如果需要与外部世界通信（如获取时间、发送 HTTP 请求），必须由宿主环境实现一个 **IO 处理器**，并将其挂载到 Agent 可访问的特定内存地址（键）。
- Agent 通过读写这些地址（例如 `MEM["$io.clock"]`）来同步或异步地获取外部数据，整个过程对 Agent 而言仍然是“信息处理”，而非“工具调用”。

## 适用场景

- 需要大量 Agent 协同完成复杂信息处理任务（如文档分析、内容生成、多轮对话编排）。
- 希望严格区分“推理”与“外部动作”的系统。
- 对 Agent 行为的可预测性、可审计性有较高要求的场景。


# 代码描述

首先我们有一个带元数据的复合数据结构：基本数据类型只有字符串，通过可以携带元数据的列表和字典来组合成更复杂的东西。这就是我们的内存模型。  
然后我们有一个对话历史数据结构（Conversation）和一个用户消息批次数据结构（UserMessageBatch），它们分别作为**对话历史寄存器**（last_msg_reg）和**用户输入批次寄存器**（usr_tool_reg）的元素。再加上一个指令栈，以及一个负责具体 LLM 调用的 LMU 模块，就构成了整个运行框架。

程序运行的核心只有两条指令：`create` 和 `execute`。

**create 指令**  
指定内存中具体的字符串作为输入，调用模型。  
- 如果模型输出里只有普通字符串（没有工具调用），就把结果直接写回目标用户输入批次寄存器——这个寄存器就是父调用的寄存器，结果作为“工具调用结果”返回。然后弹出当前指令。  
- 如果模型输出里有工具调用，就创建新的对话寄存器和新的用户输入批次寄存器，把自己（create 指令）转换成一条 `execute` 指令，再把那些工具调用压入栈中。  
值得注意：这里说的“工具调用”实际上都是虚拟机指令，而且每个调用都会附带一个返回地址，也就是本次调用对应的寄存器地址（对应上文说的“父调用的寄存器”）。  
另外，当前代码里除了 `create` 本身，其他指令的返回逻辑还没处理好，因为其他指令我还没写。

**execute 指令**  
指定两个寄存器地址作为输入，调用模型。  
- 如果输出只有字符串，处理方式同上（结果写回目标寄存器，作为工具调用结果返回），结束后会弹出自己的调用栈以及用到的两个寄存器。  
- 如果有工具调用，这一轮的调用记录会被存入自己的对话寄存器中，其他处理同上。

我们大可以想象其他各种指令，最重要的就是内存读写相关的。还可以想象中断、命令权限等机制。  
关于命令权限，有趣的一点是：对于 LLM 来说，使用这些指令是需要提示词的。只要你不给它提示词，它天然就不会越权执行命令。

我的设计里**不包括** LLM 持续和用户输入的交互。即便用户输入是固定的：先输入“做 A”，第二轮输入“做 B”，即便 LLM 面对这种固定指令可能没问题，但架构上目前没法直接支持这种事情。不过可以通过某种意义上的“虚拟化”来实现——借助复杂的内存交互和重复的单次调用，模拟出非工具调用驱动的 LLM 对话。

让我解释一个最简单的用户对话的例子（注意：其实并不是“最简单”，因为用户对话需要 IO，它只是在传统 API 调用方式里比较简单）：

1. 创建一个外部 IO 处理器，挂载到 Agent 可访问的指定内存地址。  
2. 写一段提示词，让 LLM 执行以下循环：  
   - 调用读内存指令，读取输入地址。指令会返回用户的输入。  
   - 调用写内存指令，把 LLM 自己的输出写入输出地址。指令会返回是否成功。  
   - （你也可以让 LLM 一次输出两个工具调用，也许更省 token。但要小心顺序：虚拟机对多指令的压栈是反的——LLM 最先输出的指令会最后被压入栈，因此最先执行。）  
3. 把这个提示词预先写入内存。  
4. 调用 `create` 指令，输入这个提示词和一条初始用户指令（比如“开始循环”）。
