# Agent 通用运行框架

## 核心理念

本项目并非传统意义上的“Agent 使用外部工具进行交互”的框架。我们定义的 Agent 是：

> **大部分时间内只处理信息，而不直接操作工具**。

Agent 的核心能力是**思考**，然后才可能行动。因此，框架的设计围绕以下几个简单直觉展开：

1. **Agent 需要思考才能行动** – 任何外部输出都必须经过 LLM 的推理，没有硬编码的“工具调用”捷径。
2. **Agent 应该有简单的长期记忆** – 基于内部键值存储（KV）的访问模型，不依赖外部文件系统或数据库。
3. **Agent 可以独立完成任务，但框架的真正价值在于管理大量 Agent 的协作** – 有序、可控、可扩展。

## 设计原则

- **与特定编程语言解耦** – Agent 的行为描述不绑定任何具体语言，只依赖对 LLM 通用模式的抽象。
- **几乎没有“工具”抽象** – 我们不把外部能力包装成“工具”。系统只保留对 LLM 原生能力（文本生成、结构化输出）的抽象。
- **保持 LLM “语言运算”的纯洁性** – 框架本身不内置任何工具支持（如文件读写、网络请求、Shell 执行）。所有外部交互必须通过 **IO 接口** 进行，类似于 CPU 的 **PIO（程序控制输入输出）** 模式：Agent 通过读写特定的内存映射区域来触发外部操作，而不是直接调用工具。

## 与外部系统的交互

- 本项目不提供任何自带工具（如搜索、计算器、数据库）。
- 如果需要与外部世界通信（如获取时间、发送 HTTP 请求），必须由宿主环境实现一个 **IO 处理器**，并将其挂载到 Agent 可访问的特定内存地址（键）。
- Agent 通过读写这些地址（例如 `MEM["$io.clock"]`）来同步或异步地获取外部数据，整个过程对 Agent 而言仍然是“信息处理”，而非“工具调用”。

## 适用场景

- 需要大量 Agent 协同完成复杂信息处理任务（如文档分析、内容生成、多轮对话编排）。
- 希望严格区分“推理”与“外部动作”的系统。
- 对 Agent 行为的可预测性、可审计性有较高要求的场景。


# 代码描述

首先我们有一个带有元数据的符合数据结构，基本数据类型只有字符串，通过带元数据的列表和字典复合，这是我们的内存模型。然后我们有一个对话历史数据结构和用户消息批次数据结构，分别做为对话历史寄存器和用户输入批次寄存器的元素，以及指令栈，最后我们有一个LMU处理具体的LLM调用。

程序的运行的核心在于两条指令：create和execute。

create指令指定内存中具体的字符串做为输入，调用模型，如果输出中只有字符串则直接将结果写回目标用户输入批次寄存器（也就是父调用的寄存器，做为工具调用结果返回），弹出指令。如果有工具调用，则创建对话寄存器和用户输入批次寄存器，将自己转换为execute指令，将工具调用压入栈。值得注意的是，工具调用实际上都是虚拟机指令，而且会附加一个返回地址，也就是这个调用的寄存器地址（对应上文）。此外，当前代码里没能很好处理指令返回逻辑，除了create这个指令本身。因为其他指令我还没写。

execute指令指定两个寄存器地址做为输入，调用模型，如果输出中只有字符串，处理参见上文，结束后会弹出自己的调用栈和两个寄存器。如果有工具调用，该轮次的调用记录会被记录到自己的对话寄存器中，处理参见上文。

我们大可以想象其他各种指令，最重要的就是内存读写相关。我们可以想象中断，命令权限等机制。这里的命令权限机制，有趣的一点是，因为对于LLM来说，使用这些指令是需要提示词的，只要不给提示词，LLM天然就不会越权执行命令。

我的设计里不包括LLM持续和用户输入的交互，即使假如用户输入是固定的“做A”然后“做B”，即使LLM面对这种固定指令可能没有问题，由于架构问题这里也没法支持这种事情。但是可以通过某种意义上的虚拟化来实现，通过复杂的内存交互和重复的单次调用模拟非工具调用驱动的LLM对话。

让我来解释一下一个最简单的用户对话的例子：
当然，实际上并不是最简单的，因为用户对话是需要IO的。它只是在传统API中最简单
1. 创建外部IO处理器，并挂载到Agent可访问的指定内存地址。
2. 写提示词，让LLM执行以下循环：
    1. 调用读内存指令，读取输入地址。指令会返回用户的输入
    2. 调用写内存指令，将LLM输出写入输出地址。指令会返回是否成功。
- 你也可以让LLM一次输出两个工具调用。也许会更省token？但要注意顺序。虚拟机对于多指令的压栈是反的。LLM最先输出的指令会最后压入，最先执行。
3. 将该提示词预写入内存。
4. 调用create指令，输入该提示词和“开始循环”之类的初始用户指令。
