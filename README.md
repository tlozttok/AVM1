# AVM — Agent 虚拟机

**一个以 LLM 推理为原始计算单元的底层运行时。你的程序以自然语言活在上下文里，AVM 负责让它持续运行。**

## 为什么需要 AVM

当前市面上的 Agent 框架——无论是 LangChain、AutoGen，还是 OpenClaw、Hermes——本质上都在做同一件事：**用 Python 写控制流，LLM 只是被调用的函数**。

```python
while True:
    user_input = get_input()
    response = llm.chat(user_input)
    print(response)
```

这个循环在 Python 里。LLM 不知道自己在一个循环里，不知道上下文会被反复重启，不知道自己的思考需要压缩进聊天记录。所有编排逻辑——分支判断、异常处理、状态管理、任务路由——都由人用硬代码完成。**90% 的框架代码在管理 LLM，而不是服务于任务。**

AVM 回答了一个不同的问题：

> **如果 LLM 是计算的主体，而不是被调用的函数，它的运行时应该长什么样？**

## 核心洞见

**LLM 的上下文不是一段聊天记录，而是一个可执行的自然语言程序。** 这个程序同时包含逻辑（告诉 LLM 怎么做）和数据（告诉 LLM 这是什么），两者无法也不需要分离。AVM 提供这个程序持续运行所需的全部机器逻辑。

**另一个关键洞见**：Python 和 LLM 在 AVM 的指令空间中**没有层级差异**。两者都能发起指令、响应指令。LLM 调用 Python 做精确计算，Python 在执行中遇到语义模糊时也能直接输出指令启动 LLM 对话。工具不再是"被调用的死函数"，而是和对等的执行单元。

---

## AVM 是什么

AVM 不是 Agent 框架。AVM 是一个**指令分发器 + 内存系统 + 设备总线**。

| 组件 | 做什么 | 不做什么 |
|------|--------|----------|
| **指令集** (create, memory_read/write/make) | 接收 LLM 输出的工具调用，执行对应的机器操作 | 不包含智能，不做任何推理 |
| **内存系统** ($MEM) | 全局可见的键值树，带元数据的字典/列表/字符串 | 不预设任何结构，程序自行组织 |
| **设备系统** (MemoryDevice) | 挂载外部 IO 到内存路径（用户输入、文件、网络） | 不只提供工具，设备可以主动触发指令 |
| **LMU** | 封装 OpenAI API 调用，解析 tool_calls | 不编排对话，不管理状态 |

程序逻辑全部以自然语言形式存在于 LLM 上下文中。LLM 自主决定何时读内存、何时写内存、何时启动子对话、何时结束。AVM 只是忠实地执行这些决定。

## 递归子调用：思考不被线性上下文压扁

AVM 最核心的机制是**递归子调用**：

1. `create` 指令从一段 system prompt 和 user prompt 启动全新对话
2. LLM 在处理中可以通过工具调用 (create_cmd) 启动子对话，指定子对话的提示词和初始数据
3. 子对话独立运行，有自己的上下文窗口
4. 子对话的结果通过工具调用返回值回传给父对话
5. 父对话可以启动多个子对话，子对话可以再启动孙对话

**这解决了什么问题？**

传统 LLM 对话中，所有思考被压扁在一条线性时间线里。回溯前面的推论需要翻越长长的上下文，信号衰减不可避免。复杂任务的推理质量随着深度指数下降。

AVM 的递归子调用保持**树状结构**：每个子对话有干净的上下文，有明确的任务边界。父对话只看到子对话的结论，不需要遍历其完整推理过程。深度加一，回溯成本不涨。

这就是 AVM 的**无尺度性**：复杂任务可以无限拆分，推理质量不因深度而衰减。

## 内存模型：代码数据嵌合体

AVM 的内存是一个全局可见的键值树，由三种类型组成：

- **MetaDict** — 带元数据的字典。LLM 读取时默认看到键列表（或手动设置的元数据），需要深入访问时逐个键展开
- **MetaList** — 带元数据的列表。默认看到长度和元数据
- **String** — 字符串。支持 `$` 和 `&` 前缀的地址解析

引用语法：
- `$MEM.path.to.key` — 递归解引用到最终值
- `&MEM.path.to.key` — 只解引用一次，返回字面值
- 不以 `$` 或 `&` 开头 — 视为字面值，直接返回

元数据机制意味着可以**控制 LLM 看到什么**。一个巨大的知识库可以先展示成 "知识库[keys=300]"，LLM 根据需求逐个键展开，避免上下文被无用信息淹没。

**程序和数据共用同一个地址空间。** 系统提示词存在 `$MEM.system`，用户消息存在 `$MEM.user`，中间结果存在 `$MEM.workspace`。LLM 可以读取这些路径，也可以修改它们。程序在运行中可以修改自己的"源代码"。

|------|---------|---------|
| `create` | 系统启动 或 LLM 调用 create_cmd | 发起全新的 LLM 对话 |

## 运行一个"与用户对话"的程序

`programs/deepseek_simplified.json` 是一个**完全用自然语言写成的循环对话程序**。它不是 `main.py` 里的硬编码逻辑，而是一段存在于内存中、LLM 自主执行的"代码"：

```
你是初始对话。你的任务不只是思考。你需要执行以下循环：
1. 查看用户输入和你之前的输出。
2. 处理用户数据，启动系统的内心思考...
3. 读取输入 $MEM.inputs.-1，将思考信息记录到内存，启动子思考...
4. 思考结束后，将输出写入 $MEM.outputs...
5. 每次写入 $MEM.outputs 后，再次读取 $MEM.inputs 的最后一个值...
```

`main.py` 只做两件事：初始化内存 + 压入第一条 `create` 指令。之后的一切——读取用户输入、思考、启动子对话、写入输出、重新启动自己——全部由这段自然语言程序自主控制。

**对话的控制循环不在 Python 里。它在 LLM 的上下文里。**

## 设计哲学

**控制流属于程序，不属于运行时。** 传统框架把 while 循环、if 判断、异常处理都写在 Python 里。AVM 只提供原子指令，让 LLM 自己决定何时调用哪个指令。这不仅是架构简化，更是正确性的根本改善——程序获得了语义弹性，能自主应对意外情况，而不是被硬编码的分支限制。

**对话是状态，不是过程。** 对话历史 (Conversation) 是可以在内存中断点续传的一等公民（计划实现）。一个对话可以暂停，另一个对话可以读写它的消息。这为异步、并发、定时触发提供了统一的基础。

**Python 和 LLM 对等。** 遵循 AVM 指令规范的 Python 函数成为可以主动发起指令的执行单元。计算器工具执行中遇到不确定，可以直接输出 create 指令启动 LLM 分析。两者互相外包弱点，形成无固定层级的协作网络。

## 与现有 Agent 框架的对比

| | LangChain / CrewAI / AutoGen | AVM |
|---|---|---|
| 控制流在哪里 | Python 代码（硬编码） | LLM 上下文（自然语言程序） |
| 对话模型 | 单一线性历史 | 递归嵌套子对话 |
| 状态管理 | 框架代码维护 | $MEM 全局内存 |
| LLM 的角色 | 被调用的函数 | 自主决策的执行主体 |
| 扩展方式 | 写 Python 插件 | 挂载设备到内存路径 |
| 多 Agent 协作 | 框架编排 | 通过共享内存 + 子调用自主协作 |
| 平台代码量 | ~90% 是框架逻辑 | 极简运行时 |

**AVM 不是"更好的 Agent 框架"，而是"LLM 计算的冯·诺依曼时刻"——把程序从硬编码的控制流中解放出来，让 LLM 成为真正的计算单元。**
